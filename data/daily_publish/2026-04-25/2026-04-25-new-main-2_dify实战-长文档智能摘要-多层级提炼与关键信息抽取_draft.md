# [Dify实战] 长文档智能摘要：多层级提炼与关键信息抽取

> 账号定位：技术小甜甜（new-main）
> 今日目标：revenue
> 专栏/系列：AI实践-Dify专栏

只要你真的把 Dify 用进业务场景，就一定会遇到“长文档”这个问题。

比如：

- 一份 60 页的行业报告，要快速提炼出可执行结论
- 一份客户会议纪要，要抽出待办事项、负责人和风险点
- 一组产品资料，要整理成销售可直接复述的摘要版本
- 一批知识库文档，要抽出字段后进入后续流程

很多人第一反应是：把整篇文档扔给模型，让它“帮我总结一下”。

结果往往不理想：

- 关键结论丢了
- 摘要很像“阅读理解标准答案”，但没有业务价值
- 不同次生成结果波动很大
- 一旦文档更长一点，摘要就开始失真

这不是 Dify 不行，也不是模型不行，而是“长文档摘要”本来就不应该被当成一句 prompt 能解决的问题。

真正可用的长文档智能摘要，至少要解决三件事：

1. 怎么分段，避免一口气吞太多内容
2. 怎么分层，总结既保留细节又不丢全局
3. 怎么抽字段，把结果变成结构化输出，而不只是几段空泛结论

这篇文章就从实操角度拆开讲：如何在 Dify 里做一个稳定、可复用、适合知识库和报告场景的长文档摘要工作流。

## 一、为什么普通摘要在长文档场景里经常失真或漏重点

先说结论：长文档摘要最常见的问题，不是“摘要写得不通顺”，而是“摘要不可信”。

很多人做失败，通常有 4 个原因。

### 1）把“长文档摘要”当成一次性生成任务

如果你直接把几十页内容塞进一个节点里，让模型总结，它很容易出现两种情况：

- 只抓住前半部分内容
- 只抓住显眼措辞，忽略真正关键数据

因为模型天然擅长的是“在当前上下文里生成合理文本”，并不等于它天然适合做超长材料的稳定压缩。

### 2）没有区分“给人看的摘要”和“给系统用的提取结果”

很多业务场景并不只需要一段总结话术，还需要结构化字段，比如：

- 项目名称
- 时间范围
- 核心结论
- 风险项
- 待办事项
- 负责人

如果你只要自然语言摘要，后续还得再做一轮信息抽取，误差就会叠加。

### 3）没有定义“什么叫重点”

对技术文档来说，重点可能是架构变化；
对销售周报来说，重点可能是转化数据；
对客户纪要来说，重点可能是未决问题和下一步动作。

也就是说，摘要不是“缩短篇幅”这么简单，而是要围绕场景做重点筛选。

如果不先定义目标，模型就只能凭语言表面特征来猜重点。

### 4）没有做层级控制

长文档不是缩成一段就完事。很多时候你同时需要：

- 局部段落摘要
- 文档总摘要
- 关键字段表
- 风险与建议

这就决定了你不能只生成一个“最终摘要”，而要设计一个多层级摘要流程。

## 二、多层级摘要流程：切分、分段摘要、全局汇总

如果你想让 Dify 的长文档摘要稳定，最实用的方法就是把流程拆成三层。

### 第一层：文档切分

第一步不是让模型总结，而是先把输入切成合理片段。

切分时有两个原则：

- 尽量按自然结构切，不要机械地每 N 个字就断开
- 每个片段都应尽量保持语义完整

比如一份报告，优先按：

- 摘要
- 市场背景
- 数据分析
- 结论与建议

来切。

如果没有天然结构，再考虑按段落块、章节标题、或长度阈值拆分。

为什么这一步这么重要？

因为一旦切分粗糙，后面所有摘要都会受影响。比如把一个结论段和一个方法段强行拆开，模型就可能丢失因果关系。

### 第二层：分段摘要

每个片段单独做摘要时，不建议只是写“请总结这段内容”，而是要明确要求输出什么。

一个更稳的方式是每段输出：

- 该段主旨
- 关键事实 / 数据
- 对业务有意义的信息
- 是否存在风险 / 未决事项

例如你可以把每段摘要统一成这种结构：

```markdown
- 段落主旨：
- 关键事实：
- 业务含义：
- 风险提示：
```

这样做的好处是，后面全局汇总时，你不是在处理一堆散乱自然语言，而是在处理已经结构化过的中间结果。

### 第三层：全局汇总

有了分段摘要后，最后再做一次全局汇总。此时模型处理的不是原始长文，而是压缩过的中间层信息。

这一步你可以输出：

- 一段给人看的整体摘要
- 一份结构化字段表
- 一组后续动作建议

这就是“层级摘要”的核心价值：

- 第一层保语义
- 第二层保重点
- 第三层保全局

相比“一次性总结全文”，这种方式更稳定，也更容易调试。

## 三、如何抽取关键字段并输出结构化结果

如果你的目标不只是“读完”，而是“后续还要用”，那字段抽取一定要前置考虑。

下面给你一个很适合 Dify 工作流的思路。

### 1）先定义字段模板

以“客户会议纪要”为例，你可以先定义一个结构：

```json
{
  "meeting_topic": "",
  "participants": [],
  "core_needs": [],
  "decisions": [],
  "risks": [],
  "todo_items": [],
  "owner": [],
  "deadline": []
}
```

这一步非常关键。没有字段模板，抽取就会飘；有模板，模型才知道自己要往哪里落。

### 2）按片段做初步抽取

不要等全文摘要完再抽字段。

更稳的方式是：

- 每个片段先提取自己包含的字段
- 再在全局汇总阶段做合并、去重和补全

这样能显著降低遗漏概率，尤其适合纪要、报告、法规、SOP、知识库这类信息密度高的文档。

### 3）把“字段缺失”当成正常结果

很多人做抽取时有一个误区：字段模板里每个字段都希望模型填上内容。

其实真正稳定的系统反而应该允许：

- 有些字段为空
- 有些字段待确认
- 有些字段只能给候选值

比如文档里根本没提 deadline，就不要硬猜。

在业务场景里，“不知道”比“猜错”更值钱。

### 4）把自然语言摘要和结构化字段同时输出

推荐最终输出同时包含：

- 面向人的摘要
- 面向系统的字段

例如：

```json
{
  "summary": "本次会议重点讨论了企业知识库问答系统的上线节奏与风险点……",
  "structured_data": {
    "core_needs": ["缩短客服培训时间", "提高问答准确率"],
    "risks": ["知识库更新机制未定", "测试样本不足"],
    "todo_items": ["补充 FAQ 数据集", "确定验收指标"]
  }
}
```

这会让你的 Dify 工作流一下子从“会总结”升级到“能进入后续动作”。

## 四、一个适合知识库 / 报告场景的 Dify 工作流示例

下面给你一个非常实用的 Dify 工作流思路，适合处理长报告、项目周报、客户材料、会议纪要。

### 工作流目标

输入：一篇长文档
输出：

- 一段对外可读的摘要
- 一份结构化字段结果
- 一组后续建议

### 推荐流程

#### 节点 1：文档预处理

做的事：

- 清洗无效格式
- 按章节或长度切分
- 给每个片段编号

输出示例：

- chunk_01
- chunk_02
- chunk_03

#### 节点 2：片段摘要

对每个 chunk 执行统一提示词，输出：

- 主旨
- 关键事实
- 风险点
- 可抽字段

#### 节点 3：片段抽取合并

把多个 chunk 的抽取结果做聚合：

- 去重
- 合并相近表述
- 标记冲突项

#### 节点 4：全局汇总

基于“片段摘要 + 片段抽取”生成：

- 总摘要
- 结构化结果
- 建议动作

#### 节点 5：输出渲染

根据业务场景输出不同形式：

- 给老板看的摘要版
- 给运营看的清单版
- 给系统用的 JSON 版

这个思路的最大价值，不是节点多，而是每个节点职责清楚。出了问题时，你很容易定位：

- 是切分不合理
- 是片段摘要偏了
- 还是合并逻辑有问题

这比一个“大一统模型节点”好调太多。

## 五、长文档摘要落地时的常见坑和优化建议

### 坑 1：摘要很完整，但没有行动价值

很多模型能写出一段“看起来很像总结”的内容，但读完以后你不知道下一步怎么办。

解决方法：

- 在提示词里强制要求输出“建议动作”
- 区分“事实总结”和“动作建议”
- 对于报告类内容，至少给出 3 条可执行结论

### 坑 2：结构化字段互相冲突

例如前文写“预算约 5 万”，后文写“预算上限 8 万”，如果你只是盲合并，最后结果会混乱。

建议做法：

- 聚合时保留来源 chunk
- 对冲突字段标记为“待确认”
- 不要在无依据时自动二选一

### 坑 3：切分后丢上下文

有些信息必须结合前后文才成立，比如：

- “该方案不可行”
- “这个风险需要重点关注”

如果切分太碎，模型可能不知道“该方案”指什么。

优化建议：

- 切分时保留适度重叠
- 给每个 chunk 补充章节标题
- 对核心段落保留上一个 chunk 的简短上下文摘要

### 坑 4：全局汇总时又重新幻觉

即使前面片段处理得不错，最后全局汇总仍然可能“自作发挥”。

优化建议：

- 限制全局节点只能依据中间结果输出
- 明确禁止虚构原文未出现的信息
- 对重要字段做 schema 校验

### 坑 5：摘要结果无法复用到后续工作流

如果最终只是一段 Markdown 文本，你后面还得再拆一次。

更好的方式是从第一天开始就准备两层输出：

- 给人看的文章化内容
- 给系统用的结构化结果

这样你后面无论接知识库、审核流、任务系统，都会轻很多。

## 六、什么时候该用摘要，什么时候该用信息抽取

这两个能力经常被混着用，但在业务上差别很大。

### 适合“摘要”的场景

- 领导快速看一份报告
- 内容运营快速读完一篇长材料
- 客户先看整体结论

核心目标是：降低阅读成本。

### 适合“信息抽取”的场景

- 把纪要转成待办事项
- 把文档转成结构化知识条目
- 把合同 / 方案里的关键字段抽出来

核心目标是：支持后续处理。

### 最推荐的做法

先抽取，再摘要；或者摘要和抽取一起输出。

因为业务真正要的，通常不是“看完了”，而是“能往下走”。

## 结构化总结

- 长文档智能摘要不是一句 prompt 的问题，而是一个分层处理流程。
- 稳定结果通常来自三步：先切分，再做分段摘要，最后做全局汇总。
- 如果后续还要继续流转，摘要一定要和结构化字段抽取同时设计，不能只要一段自然语言总结。
- 在 Dify 里，最推荐的做法是把预处理、片段摘要、字段聚合、全局汇总拆成职责清楚的多个节点。
- 长文档场景最常见的坑是：切分粗暴、重点不清、字段乱猜、全局汇总幻觉，以及结果无法复用。
- 真正有业务价值的摘要系统，目标不是“写得像总结”，而是“让人更快决策、让系统继续执行”。
