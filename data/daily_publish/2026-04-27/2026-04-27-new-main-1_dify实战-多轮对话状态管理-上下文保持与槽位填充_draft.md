# [Dify实战] 多轮对话状态管理：上下文保持与槽位填充

> 账号定位：技术小甜甜（new-main）
> 今日目标：revenue
> 专栏/系列：AI实践-Dify专栏

很多人第一次用 Dify 搭助手时，都会有一种“已经能用了”的错觉：模型能回答问题、界面也能聊天、偶尔还能接一个知识库，于是就觉得一个 AI 助手已经搭好了。

但只要真的把它放进业务场景，很快就会暴露一个致命问题：它只能回答“这一句”，却做不到“连续协作”。

比如用户第一次说：“帮我整理一份客户回访计划。”第二次说：“客户是 SaaS 工具站，目标是提升试用转化。”第三次说：“再加上 7 天跟进节奏。”如果助手每次都像刚认识你一样重新开始，那它就不算一个真正的业务助手，而只是一个会聊天的输入框。

这篇文章就聚焦一个非常关键、但又经常被低估的能力：多轮对话状态管理。我们不讲抽象概念，而是直接拆开来看：

1. 上下文保持到底在保什么
2. 槽位填充为什么决定了“能不能连续做事”
3. 状态持久化在实际业务里怎么落地
4. Dify 里常见的失败场景怎么兜底
5. 怎么把这套能力继续串到工作流、知识库和后续自动化里

如果你正准备把 Dify 用在客服、表单收集、项目助理、销售跟进、知识问答这类连续任务场景，这篇文章会比“模型怎么接上去”更重要。

## 一、为什么很多 Dify 助手只能回答一句，做不到连续协作

很多 Dify 应用之所以看上去“对话正常”，但一进真实业务就崩，根因通常不在模型，而在状态设计。

你可以把一个业务型助手理解成：

- 模型负责“理解和生成”
- 工作流负责“执行步骤”
- 状态负责“记住进度”

前两个大家都容易理解，第三个最容易被忽略。

举个最常见的例子。你做一个“活动方案助手”，希望它通过多轮对话收集信息，再输出一份完整方案。用户输入可能是这样的：

- 第 1 轮：我要做一个线下沙龙
- 第 2 轮：对象是 AI 创业者
- 第 3 轮：预算 2 万
- 第 4 轮：希望能带来 100 个有效线索

如果你的应用没有明确状态，它会出现三种典型问题：

1. 记不住前文，后面回答越来越飘
2. 只记住部分信息，导致生成结果缺关键字段
3. 一旦用户插入一句“顺便帮我再想个标题”，系统就乱了节奏

这就是为什么很多人会说：“模型回答挺聪明，但一旦任务复杂一点就不稳定。”

本质上，模型并不是天然具备任务状态机的。它只是根据当前上下文预测最合理的下一段文本。你不给它稳定的状态结构，它就只能靠上下文猜。而业务协作最怕“猜”。

所以，多轮对话系统想从“能聊”升级到“能办事”，必须把状态拆清楚。

## 二、上下文保持、槽位填充、状态持久化分别解决什么问题

很多人会把这三个词混在一起，其实它们各自负责不同层面。

### 1）上下文保持：让模型知道我们现在聊到哪里

上下文保持解决的是“连续性”问题。它主要告诉模型：

- 当前任务是什么
- 前面已经确认了哪些事实
- 当前还缺哪些信息
- 这一轮用户输入应该接到哪个阶段上

如果没有上下文保持，模型只能根据当前一句话做判断，很容易把“补充条件”当成“全新问题”。例如用户说：“目标客户是中小团队，不是大企业。”这其实是在修正已有方案的适用对象，而不是开启一个新任务。

### 2）槽位填充：让任务从闲聊变成结构化收集

槽位填充解决的是“任务完整性”问题。你可以把槽位理解成完成某类任务必须收集的关键字段。

比如做“内容选题助手”，核心槽位可能是：

- 账号定位
- 目标读者
- 主题方向
- 文章目标（引流 / 信任 / 转化）
- CTA
- 为什么现在写

做“销售跟进助手”，核心槽位可能是：

- 客户类型
- 当前阶段
- 核心痛点
- 预算
- 负责人
- 下一步行动

槽位填充的价值在于：它把原本松散的对话，变成可检查、可补全、可执行的数据结构。系统不再只是“听懂了没有”，而是可以明确判断哪些信息已确认，哪些缺失，哪些缺失会阻塞下一步执行。

### 3）状态持久化：让系统跨轮、跨步骤、跨节点继续工作

状态持久化解决的是“稳定性和可恢复性”问题。上下文和槽位如果只存在当前窗口里，一旦：

- 会话过长被裁剪
- 用户中途离开又回来
- 工作流跨多个节点执行
- 系统要把结果写入外部系统

你就会发现“前面收集的信息”其实并不可靠。所以真正能落地的业务助手，一定要考虑状态持久化——把关键状态写到可读取、可更新、可回溯的位置。并不一定非要复杂数据库，很多时候结构化变量、外部表、工作流变量、甚至一个明确的 JSON 记录就够了。关键是：系统不能只靠模型记忆。

## 三、一个可复用的业务助手状态设计示例

下面给你一个很适合 Dify 场景的思路。假设我们做一个“客户需求澄清助手”，目标是在 3-5 轮对话内收集需求，然后输出一份项目 brief。

### 第一步：先定义槽位

```json
{
  "project_type": null,
  "target_user": null,
  "business_goal": null,
  "budget_range": null,
  "deadline": null,
  "must_have_features": [],
  "risk_notes": []
}
```

这里的重点不是字段多，而是字段必须“能驱动下一步”。比如没有 `business_goal`，就没法判断输出方案偏增长、降本还是提效；没有 `deadline`，就没法给执行建议分优先级。

### 第二步：给每个槽位设定填充规则

不要把所有字段都交给模型自由发挥，而应该给出明确规则：

- 能从用户原话直接抽取的，就直接填
- 有歧义的，先追问
- 没必要当前轮确认的，先留空
- 如果信息足够支撑一个版本输出，就允许先生成 V1，再引导补充

例如预算槽位：

- 用户说“预算不高”，不能直接写 5000 元
- 用户说“控制在 2 万以内”，可以归一为 `0-20000`
- 用户完全没提，保持 `null`

### 第三步：为状态增加阶段标记

除了槽位本身，我建议再加一个阶段字段：

```json
{
  "stage": "collecting",
  "missing_slots": ["business_goal", "deadline"],
  "next_action": "ask_followup"
}
```

这样做的好处是，系统永远知道自己在什么阶段。常见阶段可以分成：

- `collecting`
- `clarifying`
- `ready_to_generate`
- `needs_confirmation`
- `handoff`

这样你在 Dify 里就可以根据阶段来决定：
- 是继续问问题
- 还是输出阶段性结果
- 还是进入后续工作流

## 四、在 Dify 里怎么落地这套状态管理思路

如果你是用 Dify 搭应用，我建议按下面的方式实现，而不是一开始就拼命堆提示词。

### 1）先区分聊天应用和工作流应用的边界

如果任务目标是“持续采集信息 + 最终生成结果”，更适合：

- 前端仍然用聊天体验承接输入
- 后端把关键状态交给工作流变量或外部结构记录

不要把所有逻辑都压在一段超长系统提示词里。那样短期看起来省事，长期一定会失控。

### 2）先让模型做抽取，再让规则判断下一步

推荐一个常见链路：

1. 用户输入一段自然语言
2. 模型先抽取结构化字段
3. 系统把字段更新到当前状态
4. 规则判断是否缺关键槽位
5. 再决定下一轮是追问还是输出

也就是说：
- 模型负责“理解文本”
- 规则负责“推进任务”

这比让模型一口气包办“理解 + 判断 + 决策 + 输出”稳定得多。

### 3）为异常输入准备兜底分支

真实用户不会像测试脚本那样标准输入。他们会：
- 一次说很多条件
- 中途改需求
- 插入无关问题
- 用模糊词表达

所以状态系统至少要能处理：补充信息、修正信息、插入旁路问题。

## 五、常见失败场景与兜底方案

### 坑 1：上下文越来越长，最后反而不稳定

把历史对话全塞进去不等于上下文保持。更好的做法是：
- 原始对话可以保留
- 但给模型的核心上下文，应该是“当前任务状态摘要”

### 坑 2：槽位很多，但没有优先级

不是所有信息都同等重要。要区分：
- 阻塞型槽位
- 增强型槽位

### 坑 3：状态更新没有冲突检查

用户前面说“预算 2 万”，后面又说“其实先按 5000 试试看”，如果系统只是简单拼接，就会产生脏状态。

### 坑 4：工作流节点只传文本，不传结构

节点间应优先传结构化字段，需要给人看的地方再渲染成自然语言。

## 六、一个适合知识库 / 报告场景的状态管理模板

```json
{
  "task_name": "report_assistant",
  "stage": "collecting",
  "confirmed": {
    "document_type": "客户周报",
    "target_reader": "销售负责人",
    "core_goal": "总结本周线索质量",
    "time_range": "最近7天"
  },
  "pending": ["data_source", "output_format"],
  "derived": {
    "needs_structured_table": true,
    "needs_key_risk_section": true
  },
  "next_action": "ask_followup"
}
```

这个模板的价值在于：
- confirmed 存已确认信息
- pending 存待补信息
- derived 存规则推导出的要求
- next_action 存下一步动作

## 结构化总结

- 多轮对话能不能落地，关键不在模型会不会聊天，而在有没有稳定状态。
- 上下文保持解决“我们聊到哪里”，槽位填充解决“任务还缺什么”，状态持久化解决“系统能不能继续做下去”。
- 在 Dify 里，最稳的做法不是堆提示词，而是让模型负责抽取，让规则负责推进，让结构化状态负责承接全过程。
- 真正实用的业务助手，一定要能处理补充、修正、插入问题这三类真实输入，而不是只跑通理想脚本。
- 一旦状态结构设计清楚，你后面接工作流、知识库、外部系统都会顺很多。
