# CSDN 三流程联动规范：选题 / 发文 / 运营分析

> 目标：把当前系统从“一个大而混的内容运营流程”明确拆成 3 个可独立运行、可互相喂数、可持续吸收反馈的闭环。

## 1. 为什么要拆成 3 个流程

当前项目已经具备选题、草稿保存、发布同步、专栏资产、题库与反馈等能力。

但如果把这些能力都看成一个大流程，容易出现 3 个问题：

1. 选题和发文耦合过紧
   - 选题还没稳定，就被迫一起设计发文动作
   - 发文问题会反过来污染选题讨论

2. 运营分析输出不够可执行
   - 容易停在复盘说明
   - 不能自然转成下一轮选题和增长动作

3. 用户反馈难以被准确吸收
   - 不知道是在修正选题逻辑
   - 还是在修正发文包装/上传逻辑
   - 还是在修正增长判断与分析口径

因此，这一版明确规定：

- **选题流程**：负责形成和维护高质量题库
- **发文流程**：负责消费已批准题目并完成草稿/发布流转
- **运营分析流程**：负责把结果转成下一步可执行建议

三者彼此联动，但不混成一个流程。

---

## 2. 三流程总览

### 2.1 选题流程负责什么

选题流程只负责回答：
- 这个专栏下一批该写什么
- 哪些题值得进入题库
- 哪些题不该继续
- 哪些题需要被重写、延后或废弃

它不负责直接发文，不负责代替运营分析给出收益判断结论。

### 2.2 发文流程负责什么

发文流程只负责回答：
- 今天/这轮要从已批准题库里发哪几篇
- 如何把题目变成 prompt / draft / 审核包 / 草稿链接
- 哪一篇已经进入 drafting / used / published
- 哪一步失败、是否可重试

它不负责重新大规模想题，只消费已批准题目或人工临时指定题。

### 2.3 运营分析流程负责什么

运营分析流程只负责回答：
- 最近哪些专栏/内容方向有效
- 哪些题型带来更好的流量/收益/承接
- 哪些专栏该加码，哪些该暂停，哪些该补结构
- 下一步最值得执行的选题建议和收益增长建议是什么

它不是静态复盘，而是下一轮动作输入层。

---

## 3. 流程之间的关系

三流程的关系固定为：

1. **运营分析流程** 给出下一阶段方向与约束
2. **选题流程** 基于这些输入生成专栏级题库预览
3. 用户审核通过后，题目正式进入题库
4. **发文流程** 从已批准题库中消费题目并推进发文
5. 发文结果、发布结果、表现数据回流给 **运营分析流程**
6. 用户在任一流程中的反馈都要被单独记录并用于后续改进

即：

`运营分析 -> 选题预览 -> 用户批准 -> 入题库 -> 发文 -> 发布结果/表现 -> 运营分析`

---

## 4. 选题流程规范

## 4.1 目标

选题流程的目标不是“今天先凑几篇”，而是：

- 形成 **按专栏组织的题库资产**
- 优先补齐专栏结构空缺
- 避免重复历史题目与重复结构
- 为后续发文流程提供稳定可消费库存

## 4.2 触发条件

以下情况可触发正式选题流程：

- 用户要求为某个专栏补题
- 某专栏库存不足
- 运营分析认为某专栏值得加码
- 某专栏需要结构升级、换角度或做收益补位

## 4.3 默认工作单元

正式选题流程的默认工作单元为：

- **一次只聚焦一个目标专栏**
- **一次生成 20 个候选题**
- **先预览，后入库**

也就是说：
- 不是全账号混着来
- 不是先写再说
- 而是针对某个专栏批量形成一组高质量候选题

## 4.4 标准输入

一次正式选题流程，至少应综合以下输入：

### 第一优先级：当前已发布内容基线
这是正式选题流程的起点，而不是可选项。

必须先回答：
- 当前各个专栏已经发布了什么
- 每个专栏已经覆盖了哪些模块
- 哪些题型已经写过但仍可升级
- 哪些结构已经过密、短期不该继续堆叠

在当前项目里，建议优先使用：
- 最新 `data/intel/accounts/*_full.json`
- 最新 `data/intel/accounts/*_full.md`
- 由 full capture 派生出的 `topic_libraries` / `column` 资产

1. 专栏信息
   - 专栏定位
   - 当前角色（收益/流量/结构补位）
   - 生命周期状态
   - 当前已覆盖模块与空缺模块

2. 历史文章信息
   - 当前专栏历史文章标题
   - 同账号历史已发布文章
   - 全历史去重与相似结构去重
   - 近期内容重心是否过度集中

3. 运营分析输出（如有）
   - 最近表现好的主题簇
   - 最近不该继续加码的方向
   - 需要加强的收益承接方式
   - 需要补强的内容结构

说明：
- 运营信息是正式输入层的一部分
- 但如果当前运营信息仍稀疏，不应阻塞选题流程
- 此时应明确标注：本轮主要依据“已发布内容基线 + 专栏结构”生成，运营信号只做弱约束

4. 用户反馈资产
   - 用户明确否定过的题型
   - 用户偏好的题型、角度、标题气质
   - 过去哪些批次被通过/被拒绝

5. 题库状态
   - 当前未使用 candidate
   - 已 approved / drafting / used / published 的题
   - 已 rejected / archived 的题

## 4.5 标准输出

选题流程的标准输出不是直接入库结果，而是：

- 一个 **20 题预览批次**
- 每题包含理由、角色、风险、与专栏结构的关系
- 明确说明本批次服务于哪个专栏
- 明确说明哪些题是在补位，哪些题是在加深，哪些题是在做收益承接

对于成熟专栏，预览批次应同时覆盖两种追加方向：

1. **结构补位**
   - 看当前专栏是否还有“大块结构”没有补齐
   - 例如：部署、插件、知识库、工作流、Agent、企业集成、交付治理等大类是否缺失

2. **结构内细化**
   - 在已有结构下继续细分更具体的子方向
   - 例如：同样是工作流，可以继续细化为节点设计、调试、稳定性、异常恢复、人审卡点等

同时允许多维度交叉，而不是只按一个轴出题。常见维度包括：
- 提高认识 / 扩展视野
- 实践应用
- 解决难题 / 排障 / 避坑
- 工程化 / 交付 / 稳定性

正式选题时，也可以引入外部启发作为补充来源，例如：
- 外部高热 Dify 相关文章
- Dify 官网 / 官方文档 / 官方能力说明

但外部启发只作为补充，不应覆盖当前账号自己的已发布基线与经营约束。

## 4.6 批准规则

- 预览批次生成后，**必须先给用户看**
- 用户同意前，不正式写入可消费题库
- 用户同意后，才把题目导入/更新为正式 candidate

## 4.7 选题状态建议

选题对象至少支持以下状态：

- `unused`
- `previewed`
- `approved`
- `drafting`
- `used`
- `published`
- `rejected`
- `archived`

补充规则：
- `previewed` 表示只是出现在预览批次中
- `approved` 表示用户明确允许进入正式题库
- `drafting` 表示已被发文流程取用并进入草稿阶段
- `published` 表示已通过发布同步或发布结果确认

## 4.8 反馈吸收要求

选题流程必须能吸收以下反馈：

- 指定题不要
- 某类题整体不要
- 某类题可以保留但要换角度
- 更偏实操 / 更偏收益 / 更偏承接 / 更偏流量
- 某个专栏当前不要继续加深这个方向

吸收后要影响下一次同专栏选题，而不是只停留在聊天记录里。

## 5. 发文流程规范

## 5.1 目标

发文流程的目标不是重新决定写什么，而是：

- 从已批准题库中选取可执行题
- 快速形成可审稿的发文包
- 保存草稿 / 生成审核包 / 进入人工发布
- 在每个关键节点及时回写状态

## 5.2 输入边界

发文流程只能消费以下来源：

1. 已批准题库中的题
2. 用户明确指定的临时题
3. 运营分析中明确要求立即验证的题（仍需显式批准）

默认不应跳过选题流程直接大规模自造题。

## 5.3 标准步骤

1. 从 approved topic inventory 里选择当天题目
2. 生成 publish-day / prompt / draft
3. 进入 enqueue / run / draft-save 流程
4. 生成 review package / draft URL
5. 人工审核与手动发布
6. 发布结果同步回系统

## 5.4 状态回写规则

发文流程必须具备“**成功即回写**”原则：

- 题目一旦被选中进入发文流程：更新为 `drafting`
- 草稿成功保存：至少保持 `drafting` 或推进到 `used`
- 确认已进入真实发文链路：更新为 `used`
- 发布成功或 live sync 确认为已发布：更新为 `published`

不得等整批流程全部结束后再统一补状态。

## 5.5 异常处理

- draft-save 失败：保留失败上下文，不推进到 published
- review 拒绝：允许回退到 drafting / needs_revision 对应状态
- publish sync 未命中：保留为 used，等待后续确认
- 重跑导致重复草稿：允许存在，但要记录最新 canonical draft URL

## 5.6 发文反馈吸收要求

发文流程应吸收的反馈包括：

- 标题包装不行
- 草稿结构不行
- prompt 模板不稳定
- 上传流程易丢 draft_url
- 审核包信息不够完整

这些反馈应改进发文流程本身，而不是错误地回写成“选题失败”。

## 6. 运营分析流程规范

## 6.1 目标

运营分析流程不是只做复盘，而是要产出 **下一步可以直接执行的建议**。

至少要输出两类结果：

1. **下一步选题建议**
2. **下一步收益增长建议**

必要时还应输出：
- 专栏角色调整建议
- 生命周期调整建议
- 次日/次周资源分配建议

## 6.2 标准输入

运营分析流程应综合：

- 已发布内容历史
- 近期选题批次
- 题库消费速度
- 各专栏最近表现
- 收益/流量/点击/转化相关信号
- 用户显式反馈
- 发布成功/失败记录
- 专栏分配与组合历史

## 6.3 标准输出

运营分析至少要形成以下可执行输出：

### A. 选题建议
例如：
- Dify 专栏下一轮优先补“入口层 -> 流程层 -> 稳定落地层”三段结构
- 技术前沿专栏下一轮减少概念热点，增加“热点 -> 落地判断”型题
- 某专栏暂停泛题，转做高承接题

### B. 收益增长建议
例如：
- 哪个专栏应承担旗舰收益位
- 哪个专栏适合承担第二增长位
- 哪个免费流量专栏更适合当前作为承接入口
- 哪类 CTA 或专栏路由应被加强

### C. 运营动作建议
例如：
- 单专栏补 20 题
- 暂停某专栏 1 周
- 对某专栏做结构补位而不是继续扩量
- 优先验证某一类高转化题型

## 6.4 输出要求

运营分析输出必须满足：

- 不是纯描述
- 不是只说“表现不错/一般”
- 必须能直接变成下一步选题或经营动作

也就是说，分析结果要能被下一轮选题流程直接引用。

## 6.5 反馈吸收要求

运营分析流程必须吸收：

- 用户对判断口径的修正
- 用户对收益目标的修正
- 用户对“流量 vs 收益”平衡的修正
- 用户对专栏角色认知的修正

这些反馈应影响下一次分析建议，不只是写进备注。

---

## 7. 三流程之间的标准交接物

为了避免三个流程重新耦合，必须准备标准交接物。

### 7.1 运营分析 -> 选题流程

至少输出：
- 推荐加码/暂停的专栏
- 推荐优先做的内容模块
- 推荐避免的内容模块
- 收益增长导向建议

### 7.2 选题流程 -> 发文流程

至少输出：
- 已批准题目集合
- 题目所属专栏
- 题目角色
- 题目状态
- 建议写作顺序

### 7.3 发文流程 -> 运营分析流程

至少输出：
- 已进入 drafting / used / published 的题目
- draft_url / review package / publish result
- 成功与失败记录
- 实际发布标题与最终落地结果

---

## 8. 推荐落地顺序

当前项目的推荐落地顺序如下：

### 第一步：先稳定选题流程
- 先把“当前各专栏已发布内容”固化成正式基线
- 再建立“单专栏 20 题预览 -> 同意后入题库”的正式规范
- 明确 preview 与 approved 的边界
- 让题库先成为稳定资产
- 若存在运营信息，再把运营信息叠加为下一轮选题约束和优先级依据

### 第二步：再稳定发文流程
- 从 approved inventory 消费
- 草稿保存 / review package / publish sync 成为稳定链路
- 发布后即时回写 topic 状态

### 第三步：最后稳定运营分析流程
- 让分析输出真正成为下一轮动作输入
- 固化“选题建议 + 收益增长建议”两类结果

## 9. 当前实现映射建议

基于当前 automation-system，建议后续实现映射为：

### 9.1 选题流程
- 强化 `topic_libraries`
- 强化 `topic_usage_ledger`
- 增加正式“preview batch -> approve/import”边界
- 默认支持单专栏 20 题生成

### 9.2 发文流程
- 继续使用 `prepare-publish-day`
- 继续使用 `enqueue-markdown` / `run` / `run-batch`
- 继续使用 `review_packages`
- 增加 topic 状态即时回写保证

### 9.3 运营分析流程
- 使用 live/full capture + column allocation + publish sync + usage ledger
- 形成独立分析产物
- 输出下轮选题与收益增长动作建议

---

## 10. 本规范的执行原则

后续如果用户说：
- “继续补题”
- “继续发文”
- “做下分析”

系统应先判断这是在触发哪一个流程，而不是默认进入一个混合大流程。

触发后应优先遵守：

- 流程边界清晰
- 交接物明确
- 状态及时回写
- 反馈单独吸收

这是当前 CSDN 自主运营系统后续演进的默认工作模型。
