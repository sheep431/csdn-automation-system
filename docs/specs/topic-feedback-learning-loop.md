# 选题反馈落库与学习闭环规范

> 目标：当你对一整批选题做自然语言修改时，系统不只改眼前这一批，还要把你的反馈转成结构化记录，沉淀到项目数据里，驱动后续批次持续进化。

## 1. 闭环目标

选题反馈闭环要解决 3 件事：

1. 听懂你这次在改什么
2. 把这次修改保存成可复用的数据
3. 在下一批选题里体现出学习结果

所以系统不能只停留在“把第 2 个题删掉”这种表面动作，而要继续提炼：
- 你否掉的是哪类题
- 你偏好的切角是什么
- 你更想要哪种转化路径
- 哪些题应该长期提高或降低权重

## 2. 用户反馈的输入形式

默认支持自然语言反馈，不要求你按固定 JSON 或表单填写。

典型输入：
- 第 2 个不要，太泛了
- 第 4 个角度留着，但改成更实操的避坑文
- 以后少来纯概念题
- 这种能带专栏点击的题以后多来一点
- 更适合新号，不要丢给老号
- 标题方向是对的，但切口还是太虚

系统负责把这些话拆成结构化信号。

## 3. 反馈拆解后的结构化对象

每次收到一轮批次反馈，至少要提炼出 4 类信息：

### 3.1 批次级反馈
- 本批整体是否通过
- 用户对本批的总体评价
- 本批主要问题
- 本批主要保留方向

### 3.2 题目级反馈
- topic_number
- 原标题
- 动作：保留 / 删除 / 改写 / 延后 / 升优先级 / 降优先级
- 用户原话
- 系统理解后的改动说明

### 3.3 规则级反馈
把自然语言提炼成可以跨批次复用的规则，例如：
- 降低“纯概念题”权重
- 提高“实操避坑题”权重
- 优先选择“能自然导向专栏”的题
- 对新号优先分配“问题解决型”内容

### 3.4 学习级反馈
这是更高一层的稳定偏好，用于长期进化，例如：
- 用户偏好具体、可执行、可转化的题
- 用户不喜欢空泛、过虚、纯概念型题
- 用户更看重专栏转化而不只是表面阅读量

## 4. 数据落点

### 4.1 原始反馈
保存到：
- `data/intel/feedback/`

建议文件：
- `data/intel/feedback/topic-batch-feedback_YYYY-MM-DD_HHMMSS.md`
- `data/intel/feedback/topic-batch-feedback_YYYY-MM-DD_HHMMSS.json`

用途：
- 保存你的原话
- 保存系统的逐条理解
- 保留可回溯证据

### 4.2 批次结果
保存到：
- `data/business/topic_batches/`

建议文件：
- `topic-batch_YYYY-MM-DD_HHMMSS.md`
- `topic-batch_YYYY-MM-DD_HHMMSS.json`
- `topic-batch_YYYY-MM-DD_HHMMSS.revised.md`
- `topic-batch_YYYY-MM-DD_HHMMSS.revised.json`

用途：
- 保留原始批次
- 保留你修改后的批次
- 支持对比前后差异

### 4.3 规则沉淀
保存到：
- `data/business/learning_rules/`

建议文件：
- `topic_learning_rules.json`
- `topic_learning_history_YYYY-MM.md`

用途：
- 存长期有效的选题偏好规则
- 作为下一批生成前的优先读取输入

## 5. 系统处理流程

每次收到你对一批选题的反馈后，系统按这个顺序处理：

1. 识别你在说哪几个题
2. 判断动作类型：删、改、保留、换角度、调优先级
3. 提炼通用偏好与反偏好
4. 保存原始反馈和结构化反馈
5. 生成修订后的批次版本
6. 更新学习规则
7. 在下一批选题生成时优先读取这些规则

## 6. 系统必须保存什么

每次反馈处理后，至少保存这几部分：

- 用户原话
- 被修改的题目编号
- 每个题的改动前后差异
- 系统提炼出的规则
- 这次对后续选题的影响说明

不能只留下“已修改”这种弱信息。

## 7. 系统在下一批里怎么体现学习

下一批生成时，系统必须显式说明：
- 这次提高了哪些题型权重
- 这次降低了哪些题型权重
- 哪些调整来自你的最近反馈
- 哪些题是为了测试新的判断

也就是说，后续批次不能像失忆一样重新随机来。

## 8. 推荐反馈输出格式

当系统处理完你的反馈后，建议回传 4 段内容：

### A. 已理解的修改
- 逐条确认我听懂了什么

### B. 已更新的批次
- 给出修改后的题单

### C. 已沉淀的学习规则
- 说明新增了哪些偏好或反偏好

### D. 对下一批的影响
- 明确下一批会如何变化

## 9. 当前默认规则

从现在开始，topic batch 的用户反馈按以下原则处理：

- 你的自然语言反馈默认视为可学习信号
- 系统必须保存“原始反馈 + 结构化理解 + 修订后结果”
- 系统必须沉淀跨批次规则，而不是只改单次题单
- 系统必须在下一批生成时说明学到了什么
- 系统还必须根据流程互动自动同步 topic usage 状态：
  - 删除/否决类反馈 -> `rejected`
  - 保留/改写/升降优先级类反馈 -> `approved`
  - 明确整批通过/可用 -> 将未单独否掉的题自动记为 `approved`

## 10. 后续可接入的自动化动作

这层现在已经接入了最小可用命令：

- `apply-topic-feedback`：读取一份 topic batch JSON 和你的自然语言反馈，自动生成 feedback 记录、修订后的 batch、学习规则文件
- `mark-topic-used`：把某个批次里的某个题标记为 `approved / used / published / rejected / archived`
- `topic-usage-report`：输出当前已消费选题的汇总报告

示例：

```bash
python -m app.main apply-topic-feedback \
  --date 2026-04-19 \
  --batch-path data/business/topic_batches/topic-batch_2026-04-19_090000.json \
  --feedback "第 2 个不要，太泛了。第 3 个角度留着，但改成更实操的避坑文。以后少来纯概念题。"

python -m app.main mark-topic-used \
  --date 2026-04-20 \
  --batch-path data/business/topic_batches/topic-batch_20260420090000.json \
  --topic-number 3 \
  --status used \
  --notes "已进入实际写作队列，后续批次不要重复"
```

后续还可以继续扩成：

- 自动创建 feedback 记录文件
- 自动生成 revised batch 文件
- 自动更新 `topic_learning_rules.json`
- 自动在下次 cron 运行前读取学习规则
- 自动读取 `topic_usage_ledger.json` 并对已消费题做硬去重
- 自动输出“相比上一批学到了什么”
