# 一期边界说明（当前 CSDN 运营系统）

> 目标：把当前系统收敛成一个“可稳定使用的 MVP”，避免沿着自动化与经营建模无限扩张。

## 1. 一期要解决什么

当前一期系统只服务一个核心目标：

- 让 `技术小甜甜` 这个账号形成“日更 2 篇”的稳定经营闭环
- 让这 2 篇的选题、专栏分配、发文包、状态记录、专栏经营复盘都有正式 artifact
- 让系统减少重复判断成本，但不替代最终经营裁决

一句话：

- 一期不是要做“内容运营总中台”
- 一期是要做“技术小甜甜单账号 2 篇/日 的可解释经营闭环”

---

## 2. 一期明确纳入范围

### 2.1 账号范围

当前只默认服务：

- `技术小甜甜`

老号 `踏雪无痕老爷子` 可以保留数据与兼容接口，但不作为一期主优化对象。

### 2.2 决策范围

当前纳入：

- live account snapshot
- topic batch
- 2 篇/日 的专栏分配
- 发文包（publish day）
- 专栏生命周期状态
- 专栏组合经营复盘

### 2.3 经营对象范围

当前纳入的对象：

- 收益型/付费专栏
- 免费引流专栏
- 处于 incubating / paused / deprecated 状态的专栏

### 2.4 已正式落地的核心 artifacts

一期系统默认围绕以下 artifacts 闭环：

1. `data/intel/accounts/*_live.{json,md}`
2. `data/intel/accounts/*_full.{json,md}`
3. `data/business/topic_batches/topic-batch_*.{json,md}`
4. `data/daily_plans/YYYY-MM-DD.json`
5. `data/daily_publish/YYYY-MM-DD/publish-day_YYYY-MM-DD.{json,md}`
6. `data/business/column_allocations/daily-column-allocation_YYYY-MM-DD_<account>.{json,md}`
7. `data/business/column_portfolio/column-portfolio-review_YYYY-MM-DD_<account>.{json,md}`
8. `data/state/ledger/column/*.json`
9. `data/state/ledger/column_allocation/*.json`
10. `data/state/snapshots/YYYY-MM-DD.{json,md}`

这些产物足以支撑：

- 看今天写什么
- 看今天发到哪些专栏
- 看专栏当前处于什么经营状态
- 看最近几天的分配趋势
- 看哪些专栏应该维持、试投、暂停或停止

## 3. 一期明确不纳入范围

以下内容暂不属于一期范围，除非你后续明确要求升级阶段：

### 3.1 多账号全局协同优化

暂不做：

- 新号 / 老号 联动博弈优化
- 多账号统一流量/收益分配器
- 全局最优发文调度

### 3.2 自动发文/无人发布

暂不做：

- 无人值守自动发布
- 自动点击最终发布按钮
- 自动处理全部平台侧风险

### 3.3 精细化收益预测

暂不做：

- 专栏级精确收益预测模型
- 日级 ROI 模拟器
- 复杂转化归因系统

### 3.4 无限扩张的对象建模

暂不做：

- 为每个运营概念都新增一个对象类型
- 为每个判断都新增一个独立 artifact
- 把所有长期可能有用的信息一次性全部结构化

原则：

- 只有“没有它就无法决策”或“没有它就无法执行/复盘”的对象，才进入一期

---

## 4. 一期的系统边界

### 4.1 系统职责

当前系统负责：

- 收集事实
- 生成建议
- 形成 artifact
- 沉淀状态
- 输出复盘视图

### 4.2 系统不负责

当前系统不负责：

- 替你做最终经营判断
- 保证每个选题都一定赚钱
- 代替人工完成最终审核与发布

### 4.3 判断原则

当前系统应优先满足：

1. 能跑
2. 稳定
3. 可解释
4. 可复盘

而不是先追求：

- 极度智能
- 全局最优
- 无限可扩展

---

## 5. 一期完成标准

满足下面条件，就视为一期系统已经成立，可以先收口，不继续无限扩：

### A. 日常闭环成立

系统已经能：

- 抓取或导入 live facts
- 生成正式 topic batch
- 按“2 篇/日 + 不同专栏优先”做分配
- 生成 publish day
- 记录 column lifecycle
- 自动生成 daily-column-allocation
- 输出 column-portfolio-review

### B. 人能审、能改、能继续跑

系统已经能：

- 每一步都留 artifact
- 让人读懂为什么做这个分配
- 接受人工修改并在下一轮中吸收反馈

### C. 不追求替代经营判断

系统当前只要求做到：

- 给出建议
- 给出原因
- 给出状态与趋势
- 保留人工裁决

只要这 3 条成立，就不应继续无边界扩张一期范围。

---

## 6. 一期之后再考虑什么

只有在你实际使用几轮之后，才考虑进入下一阶段，比如：

- 老号协同经营
- 专栏新增候选池
- 专栏停更/恢复自动建议器
- 更复杂的收益预测
- 更复杂的多账号调度

也就是说：

- 一期先收口
- 二期再决定要不要扩

---

## 7. 当前建议的工作方式

从现在开始，后续改进应优先遵守：

- 先闭环，后增强
- 先收口，后扩张
- 先让 `技术小甜甜` 单账号稳定跑通，再决定是否升级成更大的运营系统
