# CSDN 自主运营系统 - 第 4 层：规则层规范

> 这一层只回答：每一步按什么标准判断通过、退回、优先或延后。

## 1. 这一层的职责

规则层是把主观判断尽量标准化。

它不改变业务目标，只负责把判断条件写清楚，让人和系统都能按同一套规则执行。

## 2. 选题评分规则

### 2.1 基础评分维度

每个题至少有三项分数：

- 价值分：1~5
- 相关分：1~5
- 成本分：1~5

其中：
- 价值越高越好
- 相关越高越好
- 成本分越高表示越容易写、越省力

### 2.2 推荐判定

优先写的题通常满足：
- 价值高
- 账号相关强
- 成本分高

### 2.3 分池规则

- A 池：高价值 + 高相关 + 高成本分，优先写
- B 池：有价值，但不急
- C 池：暂存，等待素材或场景成熟

## 3. 草稿合格规则

草稿进入审核前，至少满足：

- 标题清楚
- 主题明确
- 结构完整
- 没有大量占位符
- 可以让审核人快速判断值不值得发

如果草稿只像提纲或碎片，不进入审核。

## 4. 审核通过规则

审核通过至少满足以下之一：

- 内容表达准确
- 角度适配账号
- 对读者有实际帮助
- 结构和语气符合预期

如果审核人认为：
- 题目不对
- 角度不对
- 结构不顺
- 内容太空

则进入修改流程。

## 5. 退回修改规则

需要退回修改的情况包括：

- 主题与账号不匹配
- 内容过于空泛
- 论点不清晰
- 与历史内容重复但没有新增价值
- 不是当前可写优先级

退回时要记录原因，不能只写“重写”。

## 6. 发布规则

发布前必须满足：

- 草稿存在
- 审核已通过
- 链接可打开
- 发布动作有明确负责人

如果没有通过审核，不得默认发布。

## 7. 反馈学习规则

反馈不是“看完就过去”，而是要转成可复用规则。

每次反馈至少要记录：
- 为什么通过或拒绝
- 哪个角度更好
- 哪种表达更适合账号
- 哪类题以后应提高/降低优先级

## 8. 异常处理规则

### 8.1 题池不足

如果高优先级候选题不够：
- 补充候选题池
- 不降低标准强发

### 8.2 生成失败

如果草稿生成失败：
- 保留失败上下文
- 记录失败原因
- 不默认进入审核

### 8.3 审核超时

如果长时间无人审核：
- 保留待审状态
- 不自动假定通过

### 8.4 发布失败

如果发布失败：
- 标记失败
- 保留审核通过信息
- 允许重试

## 9. 规则层输出

这一层完成后，必须形成：

- 选题分池规则
- 草稿合格标准
- 审核通过/退回标准
- 发布前置条件
- 反馈归档规则

## 10. 当前落地接口

规则层已经开始向代码层落地，当前可用接口是：

- `score-topic`
  - 用来给题目打价值分、相关分、成本分
  - 输出：`data/rules/assessments/`

- `check-draft`
  - 用来检查草稿是否满足进入审核的最低条件
  - 输出：规则判断 JSON

- `check-publish`
  - 用来检查草稿是否满足发布前置条件
  - 输出：规则判断 JSON

- `review-rules`
  - 用来把规则层记录整理成复盘
  - 输出：`data/rules/reviews/`

这一步的作用是把“可写/可审/可发”的判断标准固定下来，减少靠临场拍脑袋。

## 11. 下一层接口

规则层完成后，下一层要回答：

- 这些对象在系统里怎么表示

- 哪些字段必须保留
- 哪些历史记录必须追踪

也就是进入第 5 层：数据与状态层。
