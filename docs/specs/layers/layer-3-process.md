# CSDN 自主运营系统 - 第 3 层：流程层规范

> 这一层只回答：事情怎么流转，从选题到发布再到反馈，步骤如何衔接。

## 1. 这一层的职责

流程层把上面的战略和业务判断，变成可执行的步骤顺序。

它要定义：
- 谁先做
- 谁后做
- 什么输入进入下一步
- 什么结果回到前一步

## 2. 主流程

主流程固定为：

1. 规划主题方向
2. 收集候选选题
3. 选题打标与分级
4. 形成周/日任务
5. 生成草稿
6. 导出审核信息
7. 人工审核
8. 发布或退回修改
9. 记录反馈
10. 用反馈修正下一轮选题和写作

## 3. 关键流程节点

### 3.1 规划主题方向

输入：
- 当前账号策略
- 当前专栏结构
- 历史内容情况
- 用户需求变化

输出：
- 当前周期内的主题范围
- 本周重点方向

### 3.2 收集候选选题

输入：
- 专栏空缺
- 历史内容延伸
- 用户常问问题
- 低成本可交付题

输出：
- 候选题列表

### 3.3 选题打标与分级

输入：
- 候选题列表

输出：
- 带标签、带评分的选题池
- A / B / C 池分类

### 3.4 形成周/日任务

输入：
- 选题池
- 当日发文量
- 账号节奏

输出：
- 周任务包
- 日任务单

### 3.5 生成草稿

输入：
- 选题任务
- 账号信息
- 专栏信息

输出：
- 可审核草稿
- 草稿 ID
- 草稿链接或本地文件

### 3.6 导出审核信息

输入：
- 草稿结果
- 任务元数据

输出：
- 审核包
- 审核消息
- 发送到飞书的内容

### 3.7 人工审核

输入：
- 审核消息
- 草稿链接

输出：
- approved
- needs_revision

### 3.8 发布或退回修改

输入：
- 审核结果

输出：
- 已发布
- 退回修改

### 3.9 记录反馈

输入：
- 审核意见
- 发布表现

输出：
- 反馈记录
- 可学习的修改点

### 3.10 修正下一轮

输入：
- 反馈记录

输出：
- 选题优先级调整
- 写作偏好调整
- 账号风格修正

## 4. 人与系统分工

### 4.1 人负责

- 判断题值不值得写
- 判断是否符合人设和账号定位
- 判断是否可以发布
- 判断修改意见是否成立

### 4.2 系统负责

- 收集候选题
- 计算分数
- 组织任务
- 生成草稿
- 导出审核包
- 记录状态
- 保留历史数据

## 5. 流程中断处理

### 5.1 选题不足

如果候选题不足：
- 先补题池
- 不强行发低质量内容

### 5.2 草稿失败

如果草稿生成失败：
- 记录失败原因
- 不进入审核
- 需要回到草稿生成前一步修正

### 5.3 审核拒绝

如果人工拒绝：
- 记录拒绝原因
- 退回修改
- 不进入发布

### 5.4 发布失败

如果发布失败：
- 标记失败
- 保留草稿和审核信息
- 便于重试

## 6. 流程层输出

完成这一层后，必须形成：

- 标准流程图
- 每一步的输入输出定义
- 每一步的人工/系统职责
- 异常分支处理规则

## 7. 当前落地接口

流程层已经开始向代码层落地，当前可用接口是：

- `plan-workflow`
  - 用来把一个业务层题目变成流程执行单和待审任务
  - 输入：账号、标题、正文、栏目、标签、可选业务单路径
  - 输出：`data/process/flows/` 与 `data/tasks/pending/`

- `review-process`
  - 用来把流程执行单整理成流程层复盘
  - 输出：`data/process/reviews/`

这两个命令的作用是把“从选题到待审任务”的过程固定下来，让业务层的选择能真正进入执行层。

## 8. 下一层接口

流程层完成后，下一层需要回答：

- 每一步到底怎么判定通过与否
- 什么分数算高
- 什么内容可以进入下一步
- 什么情况必须退回

也就是进入第 4 层：规则层。
