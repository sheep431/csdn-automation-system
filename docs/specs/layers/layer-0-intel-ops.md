# 第 0 层经营输入与决策层：数据与命令落地规范

> 目的：把经营输入层真正落成可以采集、归档、检索、复盘和驱动决策的数据结构与命令入口。

## 1. 目录结构

建议在 `automation-system/data/` 下新增如下目录：

- `data/intel/`
  - `accounts/`：账号表现快照
  - `feedback/`：用户反馈与审核反馈
  - `competitors/`：对标账号、优秀案例、竞品观察
  - `market/`：搜索趋势、平台热题、外部选题信号
  - `sales/`：专栏销售、导流、CTA、转化数据
  - `insights/`：提炼后的结论与决策建议
  - `templates/`：日/周/月采集模板

### 1.1 命名规则

建议统一使用：

- 日记录：`YYYY-MM-DD.md` 或 `YYYY-MM-DD.json`
- 月总结：`YYYY-MM.md` 或 `YYYY-MM.json`
- 主题/案例记录：`slug_YYYY-MM-DD.md`

如果内容主要是人工阅读和复盘，优先用 `.md`。
如果内容主要是后续程序读取，优先用 `.json`。

## 2. 数据分层

### 2.1 原始层

只存“看见了什么”，尽量不做过多解释。

包含：
- 原始链接
- 截图
- 原始评论
- 原始数据导出
- 原始销售截图或表格

### 2.2 结论层

只存“看完后得出了什么结论”。

包含：
- 值得借鉴的结构
- 可以迁移到自己的做法
- 适合新号/旧号的判断
- 对选题、发布、销售的影响

### 2.3 决策层

只存“下一步要怎么改”。

包含：
- 本周要加重什么
- 哪类题要降权
- 哪类内容该换账号
- 哪个 CTA 更值得测试
- 哪条变现路径更有效

## 3. 核心文件格式

### 3.1 账号表现快照

文件示例：`data/intel/accounts/2026-04-19.md`

建议字段：
- 日期
- 账号
- 发文数
- 阅读
- 收藏
- 点赞
- 评论
- 关注增量
- 专栏点击
- 专栏购买
- CTA 点击
- 备注

### 3.2 用户反馈

文件示例：`data/intel/feedback/2026-04-19.md`

建议字段：
- 来源
- 反馈类型
- 原文
- 归因
- 可行动建议
- 是否进入选题池

### 3.3 对标账号/优秀案例

文件示例：`data/intel/competitors/2026-04-19.md`

建议字段：
- 账号名
- 链接
- 主题
- 标题结构
- 切入角度
- CTA 方式
- 值得复用的点
- 不适合直接照搬的点

### 3.4 市场/趋势

文件示例：`data/intel/market/2026-04-19.md`

建议字段：
- 观察来源
- 热门主题
- 相关关键词
- 趋势判断
- 对我方题材的意义

### 3.5 销售/转化

文件示例：`data/intel/sales/2026-04-19.md`

建议字段：
- 专栏名
- 引流内容
- 点击表现
- 转化表现
- CTA 位置
- 可能优化点

### 3.6 结论/决策

文件示例：`data/intel/insights/2026-04.md`

建议字段：
- 本月结论
- 重点变化
- 需要持续强化的方向
- 需要降权的方向
- 下月实验计划

## 5. 决策使用规则

### 5.1 选题决策

当一个主题满足以下条件时，提高优先级：
- 用户反馈里频繁出现
- 对标账号表现好
- 和专栏转化链路一致
- 能自然导向专栏或购买

### 5.2 账号分配

- 新号优先承担：直接解决问题、容易转化、容易建立信任的内容
- 旧号优先承担：总结型、判断型、经验型、结构性内容

### 5.3 发布决策

发布顺序建议优先考虑：
- 最近反馈强的题
- 高转化链路中的题
- 可以测试 CTA 的题
- 可以验证新结构的题

### 5.4 销售决策

如果某类免费内容能显著带来专栏点击或购买，就要：
- 加大同类题材比例
- 提前铺垫专栏路径
- 在结构中保留转化入口

## 6. 使用频率

### 6.1 日用

- 记录当日账号表现
- 记录当天反馈
- 记录当天看到的外部参考

### 6.2 周用

- 汇总一周的经营输入
- 形成周决策卡
- 调整下周选题与发布动作

### 6.3 月用

- 汇总整月表现
- 形成月复盘
- 调整专栏、账号定位和销售策略

## 7. 与系统其他层的关系

- 对第 1 层：提供战略修正依据
- 对第 2 层：提供选题与专栏依据
- 对第 3 层：提供流程优先级
- 对第 4 层：提供规则修正依据
- 对第 5 层：提供对象化存储内容
- 对第 6 层：提供 CLI 与数据目录落点

## 8. 最小可行落地顺序

建议按这个顺序实现：

1. 先创建 `data/intel/` 目录
2. 再落日记录模板
3. 再落周/月复盘模板
4. 再加最小 CLI：`collect-intel` 和 `review-intel`
5. 最后再扩展对标、销售和趋势记录

## 9. 验收标准

这一层真正落地后，应满足：

- 能每天记录经营输入
- 能每周输出决策结论
- 能每月总结策略变化
- 能把数据直接用于选题、发布和卖专栏决策

如果没有这层，系统就会缺少“现实反馈回路”，只能按计划机械运行。
